## (a) CORRECTNESS

**F1 — HIGH / high-confidence — the U1 adaptive driver defeats the chi-square gate (fail-open pathway).**
`crates/fusion-executive/src/lib.rs:294-297` — the *only* thing that ever lowers a band's trust is a **chi-square gate rejection** (`observe(band, 0.5)`); an accept feeds `observe(band, 0.0)`. Lowering trust inflates R (`scale_variance`, band_trust.rs:118 → `base/trust`), and inflating R **shrinks NIS** on that band's subsequent observations (`nis = innovation²/(HᵀPH + variance)`, estimator lib.rs:352). Smaller NIS → the gate stops rejecting → previously-rejected outliers get **admitted** (down-weighted but applied). So a band that throws sustained rejections has its integrity gate progressively *opened*, exactly backwards. This is a genuine fail-open direction for the mean state (P stays conservative — see F8 — so it hides behind an honest-looking covariance: an H2/H6 "plausible-wrong output" contributor). Bounded (~3× R inflation at the steady state of a pure-rejection stream, since each rejection targets `trust=1/(1+4·0.5)=0.333`; TRUST_FLOOR 0.05 → 20× only under interference values the code never actually feeds), but real and undocumented. Verify: trace `observe` call sites; note the sign of the R→NIS→gate feedback.

**F2 — HIGH / high-confidence — U1 provides ~zero protection against the stated threat.** The design narrative (COMBINED_SOFTWARE_BRIEF §U1 "the Black Sea survival change") is that a jammer down-weights Ku. But the wiring only reacts to gate-*failing* observations. A jammer/spoofer producing Doppler consistent enough to pass the chi-square gate feeds `observe(band, 0.0)` → trust stays 1.0 → the biased measurement is fully weighted. Fully-jammed bands produce no detections → `observe` never called → trust stays 1.0. So U1 as wired does nothing the chi-square gate isn't already doing, and its main runtime effect is the F1 feedback loop. The CODE_CHANGE_PLAN itself (§U1b) mandates a *defined statistic* — "residual/noise-floor inflation vs a clear-sky baseline… `best_quality` alone confounds range/elevation/antenna with jamming, so it is not used raw." The shipped code uses gate-rejection, which is *more* confounded than best_quality (rejections come from bad ephemeris, geometry outliers, multipath, maneuvers, wrong-Jacobian). **The implementation contradicts the author's own reviewed plan.** Verify: CODE_CHANGE_PLAN.md:82-86 vs lib.rs:279-297.

**F3 — MEDIUM — the "VHF carries" story is inert; composes with D10 only by being dead.** `Band::Vhf` maps from Orbcomm only (pnt-types lib.rs:88). Orbcomm is D10-blocked at routing (`lib.rs:150-154`, empty routing → never reaches `process_doppler`). So the VHF trust slot is never driven and "VHF (Orbcomm) survives when Ku is jammed" (brief §U1) **cannot occur in current code**. Positive side: U1 does *not* re-admit Orbcomm — D10 is respected. But the headline capability is fictional until D10 is lifted. Verify: routing match arm vs Band mapping.

**F4 — MEDIUM (latent) — U4a can mask a persistent divergence and breaks a filter invariant.** `estimator lib.rs:358-363`: with the robust gate on, `accepted=true` no longer implies `nis ≤ threshold` — a downstream check relying on "accepted ⟹ within gate" is now wrong. More importantly, the hard gate's safety role is *reject-and-coast* on biased Doppler (SAFETY_CASE §2.3 rows "Frequency reference degrades / sustained heel → reject biased Doppler" and "Sea-surface multipath → reject/inflate"; H8). The robust cost converts that into *slow-admit*: a persistent bias with reduced-but-nonzero gain still integrates into the state over many epochs while NIS stays large (and reported). Default-off today (see F8), but the plan flags U4a as a "safe starting point" to enable — that enablement would weaken H8's mitigation and must be re-reviewed against the safety case, not just unit-tested. Verify: SAFETY_CASE.md:344-345 & H8 vs the accept-branch semantics.

**F5 — LOW — Huber inflation is boundary-exact only.** `R_eff = R·(nis/threshold)` (lib.rs:360) makes the effective normalized innovation equal `threshold` **only when `HᵀPH ≪ R`**; it ignores the `HᵀPH` term, so when the state is well-known the outlier is under-down-weighted. Also the Huber knee is placed at the *gate* (e.g. NIS=4, ~2σ) rather than the canonical ~1.35σ — it's a "soft gate," not textbook Huber. Internally self-consistent (the `w=√(threshold/nis)` comment checks out) and the covariance uses the same `effective_variance` (conservative, correct direction), so not a bug — but the "correct Huber implementation" framing is loose. Verify: algebra in lib.rs:354-372.

**F6 — LOW — "unknown ⇒ trust unchanged" contradicts the plan's "unknown ⇒ 1.0."** `band_trust.rs:73-77` returns early on NaN/negative, leaving an established low-trust state *sticky*. CODE_CHANGE_PLAN §U1b says "Unknown / insufficient samples ⇒ trust = 1.0." The docstring openly documents the deviation. Safer direction (persist caution), but a stated-contract divergence to flag.
